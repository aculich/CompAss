2.4 Data Acquisition: ORACC

ORACC data are made available for analysis in JSON format under the CC0 license (public domain). JSON (JavaScript Object Notation) is a file format that is used widely for exchanging data. The file format is simple and straightforward, but allows for complex representations of data in a hierarchical format (not unlike XML). The structure of JSON files is very easy to read and parse in Python.

A typical ORACC JSON file may start like this (abbreviated):

 
{
"type": "catalogue",
"project": "dcclt",
"source": "http://oracc.org/dcclt",
"license": "This data is released under the CC0 license",
"license-url": "https://creativecommons.org/publicdomain/zero/1.0/",
"more-info": "http://oracc.org/doc/opendata/",
"UTC-timestamp": "2017-06-21T22:02:40",
"members": {"P000001": {
        "project": "dcclt",
        "author": "Englund, Robert K. &amp; Nissen, Hans J.",
        "collection": "Vorderasiatisches Museum, Berlin, Germany",
        "designation": "W 06435,a",
        "excavation_no": "W 06435,a",
        "id_text": "P000001",
        "museum_no": "VAT 01533",
        "period": "Uruk III",
        "primary_publication": "ATU 3, pl. 011, W 6435,a",
        "provenience": "Uruk",
        "publication_date": "1993",
        "publication_history": "ATU 1, 594",
        "subgenre": "Archaic Lu A",
                    }, ...
                }
}

A JSON object contains keys and values, such as "type":"catalogue", "project": "dcclt". The key is separated from its value by a colon; the key/value pairs are separated by commas.

The most important key here is "members". The value of this key is the entire set of text IDs in the DCCLT corpus. Each of the member nodes (text IDs) includes the actual catalog information. Shown here is an abbreviated version of the catalog data of the Archaic Lu A exemplar P000001, more commonly known as VAT 1533 or W 6435,a.

2.4.1 JSON Data Format

The catalog data displayed above was originally entered in Filemaker, then exported to XML for use by the ORACC software tools. One could easily transform either of those formats into .txt or .csv files, which are easier to read for the human eye. Why, then, produce JSON files? JSON is recognized as a lightweight but very versatile data structure in particular for exchanging data between programs. Databases (and .csv files) need a fixed number of fields; key/value combinations in JSON can be extended at will. Representation of hierarchical structures is very natural in JSON, but is impossible in traditional databases. We will see that ORACC JSON documents make extensive use of this feature. The two features mentioned here (extensibility and hierarchical structure) are shared with XML, which is in many ways similar to JSON. Generally, JSON is considered to be lighter (smaller files) and more efficient, because the data structure is very closely aligned to data structures in common programming languages such as Java, Python, and R.

JSON files consist entirely of "key" : "value" pairs. Keys are always strings. Values may be of the following types:

JSONPythonSurrounded byDefined as
objectdictionary{}unordered sequence of name/value pairs
arraylist[]ordered sequence of values
valueobjectstring, array, or object

A JSON object (Python dictionary) may look like this: ​ {"sex": "male", "name": "John"}

The contents of a valid JSON file are always wrapped in curly brackets, defining it as a JSON object.

A JSON array (Python list) may look like this:

["Johan", "Johnny"]

This array might be included in our JSON object as follows: ​ 

 
    {
    "sex": "male"
    "name": "John"
    "nicknames": ["Johan", "Johnny"]
    }

An array is thus a way to give multiple values to the same name or key. In the example here the values inside the array are strings (surrounded by quotation marks), but they may also be arrays or JSON objects. This allows for very complex trees with a minimal arsenal of data structures.

For a more formal and exhaustive description of the JSON data structure see http://www.json.org/.

2.4.2 Acquiring and parsing catalogue.json

The file catalogue.json is made available for every ORACC project at the address http://oracc.org/[PROJECT]/catalogue.json. If a project has subprojects, then each subproject has its own set of JSON files, including catalogue.json. Since the structure of catalogue.json is simple and there is relatively little depth in its hierarchy, it can be parsed in just a few lines with the requests library. This library is used to read and download material from the web with the function get(url). The function json() will read the result as proper JSON data.

 
import requests
url = "http://oracc.org/dcclt/catalogue.json"
r = requests.get(url)
cat = r.json()

The variable cat will now contain the entire catalogue.json file from the DCCLT project (note that in the URL dcclt should be written in lower case).

Since the structure of the JSON file is identical to the structure of a Python dictionary, we can treat the variable cat as a dictionary. The value of the key "members" is itself a dictionary of dictionaries which may be transformed into a Pandas Dataframe for ease of viewing and manipulation.

 
import Pandas as pd 
cat = cat["members"]
df = pd.DataFrame(cat)
df

Inspecting the DataFrame shows that each text is now a column and that the catalogue fields have become rows. In other words we need to transpose the DataFrame, which we can do in the same go:

 
df = pd.DataFrame(cat).T
df

The table in df now contains all the catalog data available in DCCLT. The catalogue derives from (but is not identical to) a CDLI catalogue, and thus one will find the well-known CDLI field names. The Pandas library allows one to manipulate and slice the DataFrame in many different ways. For instance, one may select the relevant fields by creating a new DataFrame as follows:

 
df1 = df[["provenience", "period", "id_text"]]

Pandas is a powerful Python library that allows for many different ways of slicing and manipulating – we will see some of those in later chapters. Various introductions to Pandas may be found on the web or in VanderPlas 2016 and similar introductions.

2.4.3 Acquiring and Parsing a Text File

ORACC JSON text files include transliteration and lemmatization, as well as information on the sign level. Translation is not included. The files are found in the corpusjson directory of each project and are named after their text ID, for instance dcclt/corpusjson/P251867.json.

The acquisition works in exactly the same way as above:

 
url = "http://oracc.org/dcclt/corpusjson/P251867.json"
r = requests.get(url)
text = r.json()

The structure of the JSON file, however, is much more complex, because of the hierarchical structure of the data. A text may have one or more surfaces (obverse, reverse), each surface may have one or more columns; each column has lines; each line has words; and each word has signs.

text object
	surface
		column
			line
				word
					sign

How many of those layers are present in a particular text is impossible to predict. Some tablets have columns, others do not; most surfaces have text, but not all surfaces do. Moreover, ORACC JSON potentially also has information about sentences or other discourse units, which may or may not not align with the structure of the object in columns and lines. ​ The JSON tree for a text edition consists of a hierarchy of three types of nodes: c for a Chunk of text (a word, a line, a sentence, or an entire text); d for Discontinuity (beginning of a line, a column, or a surface; breakage; or a ruling on the tablet); and l for Lemma, containing all the lemmatization data, including data on the graphemes that write the word.

The structure may be illustrated with the beginning of P251867, an Old Babylonian 3-line lentil (beginning omitted): 

 
"cdl": [
{
  "node": "c",
  "type": "text",
  "id": "P251867.U0",

      "ref": ""

    },

                      {
                        "v": "ŋeš",
                        "id": "P251867.2.1.0"
                      }
                    ]
                  },

The first "cdl" node contains an array of three objects: two d nodes (discontinuities) for the object and the obverse, and a c node (Chunk)representing the entire text. The c node contains a new "cdl" key that indicates a discourse unit, namely the body of the text (note that Chunk text and Chunk body are identical here - but that need not be the case). Eventually, there is a nod l that contains the lemmatization data for the first word of this text. The lowest node in this tree is called "gdl" (for Grapheme Description Language), which identifies the graphemes of which each word is composed, as well as the reading and the function (syllabogram, logogram, determinative, etc.) of those graphemes.

At any level in the tree the key "cdl" (for Chunk Discontinuity Lemma) thus has as its value an array (corresponding to a list in Python). The array contains a sequence of JSON objects, each of which is a c, d, or l node. Each c node (Chunk) may contain another "cdl" key, which again contains an array – etc. The l nodes contain a key "f" which has as its value a JSON object (or a dictionary, in Python speak) that contains all the lemmatization data. In order to pull out the lemmatization data, therefore, we need to iterate through all the "cdl" keys until we encounter an "l" node, containing an "f" key. The value of the "f" key is the data we want.

In order to do this we may define a recursive function, that is, a function that calls itself to recursively inspect the value of successive layers of "cdl" keys until one encounters an "f" key:

 
def parsejson(text):
    for JSONobject in text["cdl"]:
        if "cdl" in JSONobject: 
            parsejson(JSONobject)
        if "f" in JSONobject:
            lemm_l.append(JSONobject["f"])
    return

For this code to run we need to first define the empty list lemm_l. Then the function is called with the argument text, which contains the entire JSON file, as retrieved above. The function modifies the list, filling it with a new word each time it encounters an "f" key.

 
lemm_l = []
url = "http://oracc.org/dcclt/corpusjson/P251867.json"
r = requests.get(url)
text = r.json()
parsejson(text)

The list lemm_l now contains all the lemmatization data of P251867 as edited in DCCLT.
